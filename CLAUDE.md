This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project status

pydreg is a from-scratch Python port of [dREG](https://github.com/Danko-Lab/dREG) (Danko Lab), and **is fully implemented, tested, and validated against the original** — full end-to-end inference (informative-position scanning → feature extraction → SVR scoring → RF-assisted peak calling → FDR filtering → output writing), no training functionality. On real test data, pydreg's called peaks agree with real dREG's at a 0.999728 Jaccard index. Layout:

- `src/pydreg/` — the installable package (src-layout). One module per pipeline stage: `io.py` (bigWig via pybigtools + BED/tabix/bigWig writers), `infp.py` (informative-position scan), `features.py` (multi-scale feature extraction), `models.py` (the two pretrained models — an SVR and an RF peak-splitter — downloaded from HF via `from_pretrained()`), `backend.py` (cuML → sklearn → NumPy tiered scoring dispatch), `stats.py`/`smoothing.py`/`rfsplit.py`/`peaks.py` (the statistical/peak-calling machinery, numba-jitted where it matters), `pipeline.py` + `cli.py` (orchestration + the `pydreg` CLI entry point).
- `tests/` — pytest suite (40 tests as of this writing; `uv run pytest tests/ -q`).
- `pyproject.toml` + `uv.lock` — dependency management via `uv` (`uv sync`, `uv sync --extra gpu` for the optional cuML tier). Assume `pytest`/`uv` ARE configured now; don't re-ask the user for tooling preferences.
- `docs/METHODS.md` — plain-language walkthrough of the algorithm, for users. `docs/OPTIMIZATION.md` — plain-language summary of the performance design choices, for users. Point people here first.
- `docs/PLANNING.md` — the full original architecture/design plan, written before implementation and kept as the comprehensive design record (module boundaries, algorithm specs, batching strategy, packaging decisions, every upstream R quirk and exactly why it's reproduced rather than fixed). Read this before making structural or algorithmic changes.
- `docs/PERF_LOG.md` — a comprehensive, chronological log of every performance change since the initial implementation (what was tried, what was measured, what didn't pan out and why) — append new entries here (don't just fix perf issues silently) so optimization history survives across sessions. Its own stated ground rule: every entry must produce output that's unchanged (or, where the change is inherently statistical/randomized, unchanged within the pipeline's existing noise floor) from before the change.

`_reference/dREG/` is listed in `.gitignore` (never commit it) — treat it as read-only reference material for understanding the algorithm being ported, not as code to build on top of or import from. Same for `_models/` (local staging for the one-time R model-conversion scripts in `scripts/`, not part of the installed package).

## What dREG does (the algorithm being ported)

dREG detects active transcriptional regulatory elements (TREs) — promoters and enhancers — from GRO-seq/PRO-seq nascent-transcription data. Input is a pair of strand-specific bigWig files (plus/minus strand read counts, 5'-mapped point-mode reads, unnormalized). Output is a genome-wide score (and optionally called peaks) indicating regulatory potential at each informative position. The original pipeline, in order:

1. **`get_informative_positions`** (`_reference/dREG/dREG/R/get_informative_positions.R`) — scans the genome in windows/steps over the bigWig pair and returns candidate positions that pass a minimum read-depth filter on one or both strands (AND/OR combination logic). This restricts downstream work to loci with any transcriptional signal, since most of the genome is silent.
2. **Multi-scale feature extraction** (`read_genomic_data.R` + `src/read_genomic_data.c`) — for each candidate position, reads counts from both strands within a set of nested windows of increasing size ("zoom levels"), each represented by a fixed number of bins on either side of the center (`genomic_data_model`: `window_sizes` × `half_nWindows`). This produces a fixed-length feature vector per position that captures the local transcription profile at multiple scales, logistic- or linear-scaled.
3. **Model training** (`train_svm.R`, `get_test_set.R`) — samples positive (known TRE, e.g. TF ChIP/DNase) and negative positions, extracts features via step 2, and fits an SVR/SVM (originally `e1071`/`Rgtsvm`) to regress/classify regulatory potential. Out of scope for pydreg (inference-only).
4. **Scoring** (`eval_svm.R`) — applies a trained model to features extracted at informative positions to produce a per-position dREG score.
5. **Peak calling** (`peak_calling.R`, `peak_calling_rf.R`, `peak_calling_ext.R`) — merges scored positions into broad/narrow peaks (with an FDR-style significance threshold), refined with a random-forest step for narrower peak boundaries.

`run_predict.R`/`run_predict.bsh` wire together steps 1, 2, and 4 for the legacy "score only" workflow; `run_dREG.R`/`run_dREG.bsh` wire together all five steps for the recommended "peak calling" workflow that pydreg ports. Reading those two scripts top-to-bottom is the fastest way to see how the pieces connect end-to-end before diving into any one module — `docs/PLANNING.md`'s "Algorithms to port" section is the fully-worked-out result of having done that.

A pre-trained model (`asvm.RData`) bundles both the SVM parameters and the `genomic_data_model` (zoom/window configuration) used to train it — the two must always travel together, since a model is only meaningful relative to the exact feature layout it was trained on.

## Current implementation notes

- **Models**: both pretrained models (the SVR and the RF peak-splitter) are hosted as `.safetensors`/`.safetensors.zst` on Hugging Face at `adamyhe/dREG`, loaded via `DREGModel.from_pretrained()`/`DREGPeakSplitForest.from_pretrained()` (`models.py`). `scripts/export_dreg_model.R`/`export_dreg_rf_model.R` + `scripts/pack_safetensors.py` are the one-time R→safetensors conversion tooling that produced them (not part of the installed package) — see `docs/PLANNING.md` for the conversion details and the gotchas hit along the way (e.g. the RF model's node arrays being `(n_nodes × n_trees)`, the opposite orientation from the SVR's `SV` matrix).
- **Scoring backends**: `backend.py` dispatches across cuML (GPU) → scikit-learn (CPU, selectable but never auto-chosen) → NumPy (CPU default) tiers, all computing identical math against the same pretrained SVR weights. See `docs/OPTIMIZATION.md` for why scikit-learn isn't the CPU default despite being the "obvious" choice, and why the cuML path (real, documented, but never run on actual GPU hardware — do a real smoke test before depending on it in production) was chosen over other GPU options.
- **RF peak-splitting**: `models.py`'s `DREGPeakSplitForest` reimplements the forest traversal (`src/regTree.c`'s `predictRegTree()`, read from source, not from memory) as a numba-jitted function, not `sklearn.ensemble.RandomForestRegressor` — see `docs/OPTIMIZATION.md` for why (pydreg's real call pattern is many tiny batches, exactly the shape scikit-learn's per-tree parallel dispatch overhead is worst at).
- **Peak-calling p-value** (`stats.pmv_laplace`): a multivariate-Laplace tail probability integrated via `scipy.stats`'s private QMC machinery, driven by pydreg's own small-start adaptive sampling schedule (matching R's `mvtnorm` behavior more closely than SciPy's public API does) rather than SciPy's `multivariate_normal.cdf` directly — this was the dominant cost in peak calling before optimization; see `docs/OPTIMIZATION.md` for the summary and `docs/PERF_LOG.md` for the full investigation (root causes, every benchmark, what didn't work).
- **Parallelism**: peak calling parallelizes across `ProcessPoolExecutor` workers, one independent unit of work per broad candidate peak; each worker pins itself to a single BLAS thread (the linear algebra involved is all tiny 5×5 matrices, far below where BLAS's own multithreading helps). Feature extraction/scoring parallelism is chunked-batch based (see `docs/PLANNING.md`'s "Batching" section), not process-based.
