# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project status

pydreg is a from-scratch Python port of [dREG](https://github.com/Danko-Lab/dREG) (Danko Lab), and **is now implemented and tested** — full end-to-end inference (informative-position scanning → feature extraction → SVR scoring → RF-assisted peak calling → FDR filtering → output writing), no training functionality. Layout:

- `src/pydreg/` — the installable package (src-layout). One module per pipeline stage: `io.py` (bigWig via pybigtools + BED/tabix/bigWig writers), `infp.py` (informative-position scan), `features.py` (multi-scale feature extraction), `models.py` (the two pretrained models, downloaded from HF via `from_pretrained()`), `backend.py` (cuML → sklearn → NumPy tiered scoring dispatch), `stats.py`/`smoothing.py`/`rfsplit.py`/`peaks.py` (the statistical/peak-calling machinery), `pipeline.py` + `cli.py` (orchestration + the `pydreg` CLI entry point).
- `tests/` — pytest suite (25 tests as of this writing; `uv run pytest tests/ -q`).
- `pyproject.toml` + `uv.lock` — dependency management via `uv` (`uv sync`, `uv sync --extra gpu` for the optional cuML tier). Assume `pytest`/`uv` ARE configured now; don't re-ask the user for tooling preferences.
- `docs/PLANNING.md` — the full original architecture/design plan, written before implementation and kept as the design record (module boundaries, algorithm specs, batching strategy, packaging decisions). Read this before making structural changes.
- `docs/PERF_LOG.md` — a running log of performance/vectorization/JIT work done after the initial implementation; append new entries here (don't just fix perf issues silently) so optimization history survives across sessions.

`_reference/dREG/` is listed in `.gitignore` (never commit it) — treat it as read-only reference material for understanding the algorithm being ported, not as code to build on top of or import from. Same for `_models/` (local staging for the one-time R model-conversion scripts in `scripts/`, not part of the installed package).

## What dREG does (the algorithm being ported)

dREG detects active transcriptional regulatory elements (TREs) — promoters and enhancers — from GRO-seq/PRO-seq nascent-transcription data. Input is a pair of strand-specific bigWig files (plus/minus strand read counts, 5'-mapped point-mode reads, unnormalized). Output is a genome-wide score (and optionally called peaks) indicating regulatory potential at each informative position. The original pipeline, in order:

1. **`get_informative_positions`** (`_reference/dREG/dREG/R/get_informative_positions.R`) — scans the genome in windows/steps over the bigWig pair and returns candidate positions that pass a minimum read-depth filter on one or both strands (AND/OR combination logic). This restricts downstream work to loci with any transcriptional signal, since most of the genome is silent.
2. **Multi-scale feature extraction** (`read_genomic_data.R` + `src/read_genomic_data.c`) — for each candidate position, reads counts from both strands within a set of nested windows of increasing size ("zoom levels"), each represented by a fixed number of bins on either side of the center (`genomic_data_model`: `window_sizes` × `half_nWindows`). This produces a fixed-length feature vector per position that captures the local transcription profile at multiple scales, logistic- or linear-scaled. This C routine is the performance-critical core of the original package and the most important piece to reason carefully about when porting (window/bin math, strand handling, edge-of-chromosome behavior).
3. **Model training** (`train_svm.R`, `get_test_set.R`) — samples positive (known TRE, e.g. TF ChIP/DNase) and negative positions, extracts features via step 2, and fits an SVR/SVM (originally `e1071`/`Rgtsvm`) to regress/classify regulatory potential.
4. **Scoring** (`eval_svm.R`) — applies a trained model to features extracted at informative positions to produce a per-position dREG score.
5. **Peak calling** (`peak_calling.R`, `peak_calling_rf.R`, `peak_calling_ext.R`) — merges scored positions into broad/narrow peaks (with an FDR-style significance threshold), optionally refined with a random-forest step for narrower peak boundaries.

`run_predict.R`/`run_predict.bsh` wire together steps 1, 2, and 4 for the legacy "score only" workflow; `run_dREG.R`/`run_dREG.bsh` wire together all five steps for the recommended "peak calling" workflow. Reading those two scripts top-to-bottom is the fastest way to see how the pieces connect end-to-end before diving into any one module.

A pre-trained model (`asvm.RData`) bundles both the SVM parameters and the `genomic_data_model` (zoom/window configuration) used to train it — the two must always travel together, since a model is only meaningful relative to the exact feature layout it was trained on.

## Porting considerations

- The R package depends on the `bigWig` R package (bigWig file I/O) and calls into a custom C extension for feature extraction; a Python port will need an equivalent bigWig reader (e.g. `pyBigWig`) and should decide whether to reimplement the windowing/binning logic in Python/NumPy or wrap the existing C.
- `svm()` in the original is trained as epsilon-regression SVR (continuous dREG score) or as a probabilistic C-classification SVM (`P_SVM`), depending on `svm_type`; a Python equivalent maps to `sklearn.svm.SVR` / `sklearn.svm.SVC(probability=True)`.
- Pretrained `.RData` models (e.g. `asvm.gdm.6.6M.20170828.rdata`, [Zenodo](https://zenodo.org/records/10113379)) *are* portable — confirmed by extracting and reconstructing one. The saved `asvm` object (class `"gtsvm"`, trained via Rgtsvm) is field-compatible with e1071's `"svm"` class and is just a standard RBF epsilon-SVR dual form (`SV`, `coefs`, `rho`, `gamma`, plus x/y scaling vectors) — no re-training needed, only a format-conversion step. `scripts/export_dreg_model.R` extracts these weights (base R only, no e1071/Rgtsvm/dREG package required); `scripts/pack_safetensors.py` packs them into a single `.safetensors` file (the right format for this — not the raw `.bin` dumps, and not pickle/joblib, which aren't guaranteed cross-version-safe and execute arbitrary code on load, a bad fit for a public HF file); `scripts/dreg_model.py` (`DREGModel`) reconstructs and predicts with them in plain NumPy, validated to agree with a hand-populated `sklearn.svm.SVR` (`to_sklearn_svr()`) to ~1e-9. **cuML is a viable GPU target**, via the real, documented, stable `cuml.svm.SVR.from_sklearn(sklearn_svr)` API (added in cuML 25.06, present through the 26.06 stable release; `rapidsai/cuml` PR #6778) — confirmed by reading cuML's current source: `_predict()` rebuilds its C++ model struct from the same `dual_coef_`/`support_vectors_`/`intercept_`/`support_`/`_gamma` Python attributes on every call, so `from_sklearn()`'s attribute copy is sufficient, not a partial/lossy conversion. This was verified by source inspection only, not run end-to-end — RAPIDS/cuML doesn't install on macOS (Linux+CUDA only), so there was no GPU here to test on; do a real on-GPU smoke test before depending on it. If a non-SVM-library dependency is preferred instead, the same RBF dual-sum formula batches trivially on PyTorch/CuPy tensors — see the docstring in `scripts/dreg_model.py` for the exact formula, the `_n_support`-shape-(2,) libsvm gotcha, and validation notes.
- dREG also bundles a **second, much smaller model**: a `randomForest` regression forest (`dREG/inst/extdata/rf-model-201803.RDS`, ~500 trees, ≤153 nodes each, 10 hand-engineered peak-shape features) used only in the final peak-calling stage (`peak_calling_rf.R`: `find_rf_peaks()`/`split_peak()`) to decide whether two adjacent local maxima in a broad peak should be merged or split — it runs on a handful of candidate splits per genome, not per genomic position, so unlike the SVR **it does not benefit from GPU/cuML**; plain NumPy tree traversal is already far faster than needed at this scale. `scripts/export_dreg_rf_model.R` extracts it (again base R only) and `scripts/dreg_rf_model.py` (`DREGPeakSplitForest`) reimplements the traversal read verbatim from the package's actual C source (`src/regTree.c`'s `predictRegTree()`), not from memory. Gotcha hit and fixed here: the forest's node arrays are `(n_nodes × n_trees)` in R — the *opposite* orientation from the SVR's `SV` matrix (`n_samples × n_features`) — so the SVR export's `t(x)`-then-flatten idiom is wrong for these arrays and silently produces a garbled, cyclic tree structure (it hung the Python-side loader in an infinite loop); the fix is to flatten them directly with no transpose. Couldn't cross-check against a live R `predict()` here — this machine's R toolchain can't compile the `randomForest` package (unrelated broken linker, `-lto_library` error) — so this was validated by reading the C source plus internal tree-array invariants (all-continuous predictors, output range) rather than an external ground truth; worth a real cross-check if that matters for your use case.
- Both extracted models are packaged as `.safetensors` (safe, framework-agnostic, no arbitrary code execution) and, since they compress substantially (RF: ~88% smaller; SVR: ~71% smaller at `zstd -19`), also as `.safetensors.zst` — `scripts/_safetensors_io.py` gives both `DREGModel` and `DREGPeakSplitForest` transparent support for loading either form directly. None of this — the downloaded `.rdata`/`.RDS` sources or any exported/packed model file — is checked into git (`_models/` is gitignored, same as `_reference/`); the intent is to host these on HF instead and regenerate them from source with the `export_*`/`pack_safetensors.py` scripts if needed.
- Parallelism in the R version is CPU-core batch parallelism (`snowfall`) or GPU SVM via `Rgtsvm`; neither maps 1:1 onto typical Python tooling, so the concurrency model is an open design decision rather than something to mirror literally.
